# CP Processing Service

## Описание проекта

Сервис для обработки коммерческих предложений (КП), а также предоставления API для работы с задачами обработки.

Сервис реализует асинхронную обработку загруженных данных и хранит результаты в базе данных.

# 1. Как запустить проект

## Требования

* Java 25
* Docker и Docker Compose
* Maven

## Шаги запуска

1. Сборка проекта

```powershell
mvn clean package -DskipTests
```

2. Запуск приложения и базы данных

```powershell
docker-compose up --build
```

## Доступ к сервису

* API: http://localhost:8080
* Swagger UI: http://localhost:8080/swagger-ui/index.html

# 2. Используемые технологии

* **Spring Boot** - быстрый старт и удобная экосистема для backend
* **Spring Web** - реализация REST API
* **Spring Data JPA** - удобная работа с БД
* **PostgreSQL** - надёжная реляционная СУБД
* **Spring Security** - аутентификация и авторизация
* **Spring Validation** - валидация входных данных
* **Springdoc OpenAPI (Swagger)** - документация API
* **Mockito + JUnit 5** - unit-тестирование
* **Docker + Docker Compose** - контейнеризация и простота запуска

# 3. Архитектура решения

Проект построен по классической многослойной архитектуре:

```
Controller → Service → Repository → Database
```

## Слои

* **Controller**
  Обработка HTTP-запросов и валидация входных данных

* **Service**
  Реализация бизнес-логики и асинхронной обработки задач

* **Repository**
  Работа с базой данных через JPA

* **Entity**
  Модели данных

* **DTO + Mapper**
  Изоляция API от внутренней структуры

* **Exception Handler**
  Централизованная обработка ошибок

## Асинхронная обработка

* Используется `@Async`
* Задача обрабатывается в отдельном потоке
* Статусы задач:

  * NEW
  * PROCESSING
  * DONE
  * ERROR

# 4. Принятые компромиссы

* Реальная обработка и парсинг файлов заменены на mock-логику ввиду отсутствия конкретных требований
* Используется Basic Auth вместо JWT для упрощения реализации
* Асинхронная обработка реализована через Spring без использования очередей (Kafka/RabbitMQ)
* Не реализована полноценная модель пользователей - используются in-memory заглушки
* Нет описания ошибок в Swagger UI

# 5. Как проверить работу сервиса

## Авторизация

Используется Basic Auth:

```
username: user
password: password
```

или

```
username: admin
password: admin
```

На данный момент различия в правах доступа между ролями отсутствуют.

## Основной сценарий

1. Создать задачу
2. Получить её статус
3. Проверить результат

## Проверка через Swagger UI

1. Открыть:

```
http://localhost:8080/swagger-ui/index.html
```

2. Нажать **Authorize**
3. Ввести `user/password`
4. Выбрать endpoint и нажать **Try it out**
5. Нажать **Execute**

# 6. Примеры запросов

## Swagger UI

1. Перейти по адресу:

```
http://localhost:8080/swagger-ui/index.html
```

2. Выбрать нужный endpoint и нажать **Try it out**
3. Нажать **Execute** для отправки автоматически сгенерированного запроса
4. При необходимости выполнить авторизацию
5. Для GET-запроса указать `id`, полученный из POST-запроса

## Postman

1. Создать POST-запрос
2. Указать URL:

```
http://localhost:8080/cp/tasks
```

3. В разделе **Authorization** выбрать Basic Auth:

```
username: user
password: password
```

4. В разделе **Body → raw → JSON** указать:

```json
{
  "fileName": "test.zip"
}
```

5. Для GET-запроса использовать:

```
http://localhost:8080/cp/tasks/{id}
```

## Примеры ответов

### POST

```json
{
  "id": "0a3650c9-adf2-416a-8a28-7678888d542a"
}
```

### GET

```json
{
  "id": "0a3650c9-adf2-416a-8a28-7678888d542a",
  "fileName": "test.zip",
  "status": "DONE",
  "extractedData": "{\"items\": []}",
  "error": null,
  "createdAt": "2026-04-17T10:15:41.23836",
  "updatedAt": "2026-04-17T10:15:46.306115"
}
```

### Ошибка (не найдено)

```json
{
  "error": "Task not found"
}
```

### Ошибка валидации

```json
{
  "fileName": "fileName must not be blank"
}
```
